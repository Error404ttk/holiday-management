# เอกสารออกแบบระบบจัดการวันหยุด HOSxP

## 1. ชื่อระบบ

**ระบบจัดการวันหยุด HOSxP**  
สำหรับตรวจสอบ เพิ่ม แก้ไข ปิดใช้งาน และจัดการข้อมูลวันหยุดที่ใช้งานร่วมกับระบบ HOSxP

---

## 2. วัตถุประสงค์

ระบบนี้จัดทำขึ้นเพื่อให้เจ้าหน้าที่ที่ได้รับมอบหมายสามารถจัดการข้อมูลวันหยุดของระบบ HOSxP ได้อย่างสะดวก ปลอดภัย และตรวจสอบย้อนหลังได้ โดยลดความเสี่ยงจากการแก้ไขฐานข้อมูลโดยตรง

วัตถุประสงค์หลัก ได้แก่

1. ตรวจสอบว่าวันที่ที่ระบุเป็นวันหยุดหรือไม่
2. แสดงรายการวันหยุดตามปีปฏิทินหรือปีงบประมาณ
3. เพิ่มวันหยุดใหม่เข้าสู่ระบบ
4. แก้ไขข้อมูลวันหยุดเดิม
5. ปิดใช้งานวันหยุดโดยไม่ลบข้อมูลถาวร
6. บันทึกประวัติการทำรายการทุกครั้งเพื่อใช้ตรวจสอบย้อนหลัง
7. รองรับฟอนต์ภาษาไทยและฟอนต์ราชการ
8. ลดการใช้สิทธิ์ฐานข้อมูลเกินความจำเป็น

## 3. ขอบเขตระบบ

### 3.1 สิ่งที่ระบบต้องทำได้

- Login เข้าระบบ
- แสดง Dashboard สรุปข้อมูลวันหยุด
- แสดงรายการวันหยุดแบบตาราง
- เลือกปีปฏิทินได้
- เลือกปีงบประมาณได้
- ค้นหาวันหยุดตามชื่อหรือวันที่
- กรองตามประเภทวันหยุด
- กรองตามสถานะ
- เพิ่มวันหยุด
- แก้ไขวันหยุด
- ปิดใช้งานวันหยุด
- ตรวจสอบวันหยุดรายวัน
- บันทึก Audit Log
- Export รายการเป็น Excel/PDF ในอนาคต
- รองรับฟอนต์ Sarabun / TH Sarabun New

### 3.2 สิ่งที่ระบบไม่ควรทำในเวอร์ชันแรก

- ไม่ควรลบข้อมูลถาวรจาก HOSxP
- ไม่ควรให้สิทธิ์ฐานข้อมูลแบบ `ALL PRIVILEGES`
- ไม่ควรใช้ user `root` ของฐานข้อมูล
- ไม่ควรแก้ไขตารางอื่นของ HOSxP ที่ไม่เกี่ยวข้องกับวันหยุด
- ไม่ควรเดาชื่อตาราง HOSxP หากยังไม่ได้ตรวจสอบจากฐานข้อมูลจริง

## 6. ตารางข้อมูลที่เกี่ยวข้อง

> หมายเหตุ: ห้ามเดาชื่อตารางวันหยุดของ HOSxP ให้ตรวจสอบจากฐานข้อมูลจริงก่อนทุกครั้ง

### 6.1 คำสั่งค้นหาตารางที่อาจเกี่ยวข้อง

```sql
SHOW TABLES LIKE '%holiday%';
SHOW TABLES LIKE '%holi%';
SHOW TABLES LIKE '%calendar%';
SHOW TABLES LIKE '%day%';
```

### 6.2 ค้นหาจาก information_schema

```sql
SELECT 
  TABLE_NAME,
  COLUMN_NAME
FROM information_schema.COLUMNS
WHERE TABLE_SCHEMA = DATABASE()
  AND (
    TABLE_NAME LIKE '%holiday%'
    OR COLUMN_NAME LIKE '%holiday%'
    OR COLUMN_NAME LIKE '%date%'
  )
ORDER BY TABLE_NAME, COLUMN_NAME;
```

### 6.3 ดูโครงสร้างตาราง

```sql
DESCRIBE ชื่อตาราง;
```

### 6.4 ดูข้อมูลตัวอย่าง

```sql
SELECT *
FROM ชื่อตาราง
LIMIT 20;
```

## 7. สิทธิ์ฐานข้อมูลที่แนะนำ

### 7.1 หลักการ

ให้สิทธิ์เท่าที่จำเป็นเท่านั้น

ห้ามใช้:

- root
- ALL PRIVILEGES
- GRANT OPTION
- DROP
- ALTER
- DELETE ถ้าไม่จำเป็น

### 7.2 User สำหรับระบบจริง

ตัวอย่างกรณี Web/App Server มี IP `192.168.99.50`

```sql
CREATE USER 'holiday_app'@'192.168.99.50'
IDENTIFIED BY 'StrongPasswordHere';
```

ให้สิทธิ์เฉพาะตารางวันหยุดจริงของ HOSxP

```sql
GRANT SELECT, INSERT, UPDATE
ON hosxp.ชื่อตารางวันหยุด
TO 'holiday_app'@'192.168.99.50';
```

ให้สิทธิ์กับฐานของระบบเอง

```sql
GRANT SELECT, INSERT, UPDATE
ON hosxp_holiday_app.*
TO 'holiday_app'@'192.168.99.50';

FLUSH PRIVILEGES;
```

### 7.3 สิทธิ์ตามฟีเจอร์

| ฟีเจอร์ | สิทธิ์ฐานข้อมูล |
|---|---|
| ดูรายการวันหยุด | SELECT |
| ค้นหาวันหยุด | SELECT |
| ตรวจสอบวันหยุด | SELECT |
| เพิ่มวันหยุด | INSERT |
| แก้ไขวันหยุด | UPDATE |
| ปิดใช้งานวันหยุด | UPDATE |
| ลบวันหยุด | ไม่แนะนำ |
| บันทึก Audit Log | INSERT |
| ดู Audit Log | SELECT |

## 11. การเลือกปี

ระบบต้องรองรับการเลือกปี 2 รูปแบบ

### 11.1 ปีปฏิทิน

ตัวอย่าง:

```text
ปี 2569 = 1 มกราคม 2569 - 31 ธันวาคม 2569
```

Query:

```sql
WHERE holiday_date BETWEEN '2026-01-01' AND '2026-12-31'
```

### 11.2 ปีงบประมาณ

ตัวอย่าง:

```text
ปีงบประมาณ 2569 = 1 ตุลาคม 2568 - 30 กันยายน 2569
```

Query:

```sql
WHERE holiday_date BETWEEN '2025-10-01' AND '2026-09-30'
```

### 11.3 การแสดงผลปี

- ฐานข้อมูลควรเก็บวันที่เป็น ค.ศ. เช่น `2026-05-21`
- หน้าเว็บควรแสดงเป็น พ.ศ. เช่น `21/05/2569`
- API ควรส่งวันที่เป็น ISO format เพื่อป้องกันความสับสน

## 12. Validation

### 12.1 เพิ่มวันหยุด

ต้องตรวจสอบ:

- วันที่ห้ามว่าง
- ชื่อวันหยุดห้ามว่าง
- ประเภทห้ามว่าง
- วันที่ต้องเป็นรูปแบบถูกต้อง
- ห้ามเพิ่มวันที่ซ้ำโดยไม่แจ้งเตือน
- หากพบวันที่ซ้ำ ให้แสดง Dialog ยืนยัน

ข้อความตัวอย่าง:

```text
พบวันหยุดวันที่นี้อยู่แล้ว
ต้องการแก้ไขรายการเดิมหรือยกเลิกการเพิ่มข้อมูล?
```

### 12.2 แก้ไขวันหยุด

ต้องตรวจสอบ:

## 17. Security Requirement

1. ต้องใช้ HTTPS หากใช้งานผ่านเครือข่ายที่ไม่ใช่ localhost
2. Password ต้อง hash ด้วย bcrypt หรือ argon2
3. ห้ามเก็บ password แบบ plain text
4. ต้องมี session timeout
5. ต้องมี CSRF protection หากใช้ cookie-based session
6. ต้อง validate input ทั้ง frontend และ backend
7. ต้องใช้ parameterized query เท่านั้น
8. ห้ามต่อ SQL ด้วย string ตรง ๆ จาก input ผู้ใช้
9. จำกัด IP ของ database user
10. ไม่ใช้ root database user
11. บันทึก IP และ User-Agent ใน Audit Log
12. ควรสำรองข้อมูลก่อนแก้ไขข้อมูล HOSxP จริง

## 23. ตัวอย่าง SQL สำหรับ HOSxP v3 MySQL/MariaDB ตาราง `holiday`

> อ้างอิงจากตัวอย่างที่ใช้งานกับ HOSxP v3: `SELECT * FROM holiday WHERE YEAR(holiday_date) = '2026';`  
> ก่อนใช้จริงต้องตรวจสอบโครงสร้างตาราง `holiday` ของฐานข้อมูลจริงก่อนเสมอ เพราะแต่ละระบบ/เวอร์ชันอาจมี field เพิ่มเติมหรือแตกต่างกันได้

### 23.1 ตรวจสอบว่ามีตาราง `holiday` หรือไม่

```sql
SHOW TABLES LIKE 'holiday';
```

### 23.2 ตรวจสอบโครงสร้างตาราง `holiday`

```sql
DESCRIBE holiday;
```

หรือดู field แบบละเอียดจาก `information_schema`

```sql
SELECT 
  COLUMN_NAME,
  COLUMN_TYPE,
  IS_NULLABLE,
  COLUMN_DEFAULT,
  COLUMN_KEY,
  EXTRA
FROM information_schema.COLUMNS
WHERE TABLE_SCHEMA = DATABASE()
  AND TABLE_NAME = 'holiday'
ORDER BY ORDINAL_POSITION;
```

### 23.3 ดูข้อมูลตัวอย่างจากตาราง `holiday`

```sql
SELECT *
FROM holiday
ORDER BY holiday_date DESC
LIMIT 20;
```

### 23.4 ดูรายการวันหยุดตามปี ค.ศ. 2026

ใช้ได้ตามตัวอย่างนี้

```sql
SELECT *
FROM holiday
WHERE YEAR(holiday_date) = '2026'
ORDER BY holiday_date ASC;
```

อย่างไรก็ตาม หากต้องการให้ใช้ index ได้ดีกว่า แนะนำใช้ช่วงวันที่แทน `YEAR()`

```sql
SELECT *
FROM holiday
WHERE holiday_date BETWEEN '2026-01-01' AND '2026-12-31'
ORDER BY holiday_date ASC;
```

### 23.5 ดูรายการวันหยุดตามปี พ.ศ. 2569

ปี พ.ศ. 2569 = ค.ศ. 2026

```sql
SELECT *
FROM holiday
WHERE holiday_date BETWEEN '2026-01-01' AND '2026-12-31'
ORDER BY holiday_date ASC;
```

### 23.6 ดูรายการวันหยุดตามปีงบประมาณ 2569

ปีงบประมาณ 2569 = 1 ตุลาคม 2568 ถึง 30 กันยายน 2569  
เทียบเป็น ค.ศ. คือ 2025-10-01 ถึง 2026-09-30

```sql
SELECT *
FROM holiday
WHERE holiday_date BETWEEN '2025-10-01' AND '2026-09-30'
ORDER BY holiday_date ASC;
```

### 23.7 ตรวจสอบว่าวันที่หนึ่งเป็นวันหยุดหรือไม่

```sql
SELECT *
FROM holiday
WHERE holiday_date = '2026-05-04'
LIMIT 1;
```

### 23.8 ตรวจสอบวันซ้ำก่อนเพิ่มข้อมูล

ก่อน `INSERT` ต้องเช็กก่อนเสมอ

```sql
SELECT COUNT(*) AS total
FROM holiday
WHERE holiday_date = '2026-05-04';
```

ถ้า `total > 0` ให้ระบบแจ้งเตือนผู้ใช้ว่า “วันที่นี้มีข้อมูลวันหยุดอยู่แล้ว” และไม่ควรเพิ่มซ้ำโดยอัตโนมัติ

### 23.9 ตัวอย่าง INSERT วันหยุด

> ต้องปรับชื่อ column ให้ตรงกับ schema จริงจาก `DESCRIBE holiday` ก่อนใช้จริง  
> ตัวอย่างนี้สมมติว่าตารางมี field หลักคือ `holiday_date` และ `holiday_name`

```sql
INSERT INTO holiday (
  holiday_date,
  holiday_name
) VALUES (
  '2026-05-04',
  'วันฉัตรมงคล'
);
```

ถ้าตารางจริงมี field เพิ่ม เช่น `holiday_type`, `active_status`, `note`, `update_datetime` ให้ปรับตามจริง เช่น

```sql
INSERT INTO holiday (
  holiday_date,
  holiday_name,
  holiday_type,
  active_status,
  note
) VALUES (
  '2026-05-04',
  'วันฉัตรมงคล',
  'ราชการ',
  'Y',
  'เพิ่มจากระบบจัดการวันหยุด HOSxP'
);
```

### 23.10 ตัวอย่าง UPDATE วันหยุด

> ต้องใช้เงื่อนไขที่ชัดเจน เช่น `holiday_date` หรือ primary key จริง ห้าม update แบบไม่มี `WHERE`

แก้ชื่อวันหยุดจากวันที่

```sql
UPDATE holiday
SET holiday_name = 'วันฉัตรมงคล'
WHERE holiday_date = '2026-05-04';
```

แก้ชื่อวันหยุดพร้อมหมายเหตุ ถ้าตารางมี field `note`

```sql
UPDATE holiday
SET 
  holiday_name = 'วันฉัตรมงคล',
  note = 'แก้ไขตามประกาศวันหยุดราชการล่าสุด'
WHERE holiday_date = '2026-05-04';
```

ถ้าตารางมี primary key เช่น `holiday_id` ให้ใช้ primary key จะปลอดภัยกว่า

```sql
UPDATE holiday
SET 
  holiday_date = '2026-05-04',
  holiday_name = 'วันฉัตรมงคล'
WHERE holiday_id = 123;
```

### 23.11 ตัวอย่างปิดใช้งานวันหยุดแทนการลบ

ถ้าตารางมี field สถานะ เช่น `active_status`

```sql
UPDATE holiday
SET active_status = 'N'
WHERE holiday_date = '2026-05-04';
```

ถ้าตารางมี field เช่น `is_active`

```sql
UPDATE holiday
SET is_active = 0
WHERE holiday_date = '2026-05-04';
```

หากตาราง `holiday` ไม่มี field สำหรับปิดใช้งาน ห้ามเพิ่ม field เองในฐาน HOSxP ทันที ให้พิจารณาแนวทางอย่างใดอย่างหนึ่ง:

1. ไม่ทำฟีเจอร์ปิดใช้งานใน HOSxP โดยตรง
2. ใช้การแก้ไขข้อมูลผ่านหน้าจอ HOSxP เดิม
3. เก็บสถานะปิดใช้งานไว้ในฐาน `hosxp_holiday_app` ของระบบเอง แล้วไม่แก้ schema HOSxP
4. ขออนุมัติ/ทดสอบก่อนปรับ schema จริง

### 23.12 ตัวอย่าง DELETE

ไม่แนะนำให้เปิดใช้ในระบบเวอร์ชันแรก

```sql
-- ไม่แนะนำในระบบจริง
DELETE FROM holiday
WHERE holiday_date = '2026-05-04';
```

หากจำเป็นต้องลบจริง ต้องมีอย่างน้อย:

- สำรองข้อมูลก่อนลบ
- จำกัดเฉพาะ Super Admin
- ให้ผู้ใช้กรอกเหตุผล
- บันทึก Audit Log
- แสดง Dialog ยืนยัน 2 ชั้น
- ใช้ primary key ที่ชัดเจน ไม่ใช้เงื่อนไขกว้าง

### 23.13 Transaction สำหรับเพิ่ม/แก้ไขพร้อม Audit Log

หากระบบมีฐานของตัวเอง เช่น `hosxp_holiday_app.audit_logs` ควรทำเป็น transaction

START TRANSACTION;

INSERT INTO holiday (

  'วันฉัตรมงคล'

### 23.14 Query ที่ Backend ควรใช้แบบ Parameterized

ห้ามต่อ SQL จาก input ผู้ใช้โดยตรง เช่น ห้ามทำแบบนี้

```js
// ห้ามใช้
const sql = "SELECT * FROM holiday WHERE YEAR(holiday_date) = '" + year + "'";
```

ให้ใช้ parameterized query เช่น Node.js/mysql2

```js
const [rows] = await db.execute(
  `SELECT *
   FROM holiday
   WHERE holiday_date BETWEEN ? AND ?
   ORDER BY holiday_date ASC`,
  ['2026-01-01', '2026-12-31']
);
```

หรือสำหรับ Laravel

```php
$rows = DB::table('holiday')
    ->whereBetween('holiday_date', ['2026-01-01', '2026-12-31'])
    ->orderBy('holiday_date')
    ->get();
```

### 23.15 Mapping ปี พ.ศ. เป็น ค.ศ.

Frontend อาจให้ผู้ใช้เลือกปี พ.ศ. เช่น 2569 แล้ว Backend แปลงเป็น ค.ศ.

```text
ค.ศ. = พ.ศ. - 543
2569 - 543 = 2026
```

ตัวอย่างช่วงปีปฏิทิน

```text
BE 2569 → CE 2026
start_date = 2026-01-01
end_date   = 2026-12-31
```

ตัวอย่างช่วงปีงบประมาณ

```text
Fiscal BE 2569
start_date = 2025-10-01
end_date   = 2026-09-30
```
